PREGUNTA 1 - Modelos de Licenciamiento

Respuesta Correcta: D (Copyleft GPL v3)

Justificación Técnica Detallada: La opción D es la única que satisface integralmente los requisitos del enunciado bajo un marco legal y técnico riguroso. La licencia General Public License versión 3 (GPLv3) es el estándar de oro para el software "Copyleft". Cumple con el requisito de "código fuente accesible" porque obliga a la distribución del código fuente junto con los binarios. Permite la "capacidad de modificación interna" ya que otorga libertad al usuario de estudiar y alterar el programa. La clave distintiva es la "obligación legal de mantener la misma licencia en obras derivadas": esto es el principio del Copyleft, diseñado para garantizar que las mejoras realizadas sobre un software libre sigan siendo libres, evitando su privatización por terceros. Finalmente, se distribuye bajo términos gratuitos (gratuidad en el uso y redistribución, aunque no necesariamente exenta de costes de servicio).

Análisis de Distractores: * A) Propietaria: Es incorrecta porque las licencias propietarias (EULA) restringen explícitamente la modificación interna y el acceso al código fuente. El usuario solo adquiere una licencia de uso, no de propiedad intelectual sobre el código. * B) Freeware: Aunque es "distribución gratuita", el término freeware se refiere a software gratuito pero con código cerrado (proprietario). No cumple con la exigencia de acceso al código fuente ni permite modificaciones internas legales. * C) Open Source (MIT): La licencia MIT es permisiva y sí permite acceso al código y modificación, pero NO obliga a mantener la misma licencia en obras derivadas. Si usas una librería MIT en tu software comercial, puedes distribuir ese software como propietario sin liberar su código fuente. Por tanto, falla el criterio de "obligación legal de mantener la misma licencia".

PREGUNTA 2 - Protocolos de Correo Electrónico

Respuesta Correcta: C (IMAP4)

Justificación Técnica Detallada: El protocolo IMAP4 (Internet Message Access Protocol versión 4) es el estándar arquitectónico para la sincronización en entornos multi-dispositivo. A diferencia de otros protocolos, IMAP mantiene el estado del buzón centralizado en el servidor. Esto significa que las acciones realizadas en un cliente (marcar como leído, mover a carpeta, borrar) se reflejan inmediatamente en todos los demás clientes conectados al mismo buzón. El requisito de "eliminar mensajes del cliente sin afectar su presencia en el servidor" se refiere a la capacidad de gestionar vistas locales (como carpetas de borrado) mientras el servidor mantiene una copia persistente hasta que se decide eliminarla definitivamente desde cualquier punto, o mediante sincronización de estados.

Análisis de Distractores: * A) POP3: Es incorrecto porque su modelo es de descarga ("Download and Delete"). Tradicionalmente, POP3 descarga los mensajes al cliente local y elimina la copia del servidor para ahorrar espacio. Esto impide el acceso simultáneo consistente desde múltiples dispositivos; si borras un email en tu móvil, seguirá existiendo en tu ordenador (o viceversa) dependiendo de la configuración de retención. * B) SMTP: Es incorrecto porque Simple Mail Transfer Protocol es exclusivamente para el envío de correos del cliente al servidor o entre servidores. No tiene capacidad de gestión, lectura o almacenamiento de buzón por parte del usuario final. * D) MIME: Es incorrecto porque Multipurpose Internet Mail Extensions no es un protocolo de transporte ni acceso, sino un estándar de formato para codificar el contenido (imágenes, adjuntos, texto multilingüe) dentro de los mensajes de correo.

PREGUNTA 3 - Transferencia Segura de Ficheros

Respuesta Correcta: B (SFTP sobre SSH)

Justificación Técnica Detallada: La opción B es la solución técnica óptima para restricciones de firewall modernas basadas en seguridad. SFTP (SSH File Transfer Protocol) encapsula toda la comunicación dentro de un túnel cifrado SSH, que opera nativamente en el puerto 22. Al utilizar este protocolo, se cumple con el requisito del firewall restrictivo ("solo permite tráfico cifrado en puerto 22") y garantiza la confidencialidad e integridad de los datos mediante criptografía asimétrica/simétrica durante la transmisión. No requiere abrir puertos adicionales ni exponer canales de control y datos separados.

Análisis de Distractores: * A) FTP tradicional: Es incorrecto porque utiliza el puerto 21 para comandos y el 20 para datos, operando en texto plano (sin cifrar). Un firewall corporativo restrictivo bloquearía esto por defecto debido al riesgo de interceptación de credenciales y datos. * C) HTTP/HTTPS genérico: Aunque HTTPS es seguro, no está diseñado específicamente para la transferencia interactiva de directorios complejos como FTP/SFTP. Además, muchas políticas de firewall corporativo restringen el tráfico web a puertos específicos (80/443) y pueden bloquear el uso de HTTP para transferencias de archivos por motivos de seguridad o control de ancho de banda. * D) TFTP: Es incorrecto porque Trivial FTP no soporta autenticación ni cifrado alguno. Su diseño es minimalista para entornos de arranque en red (PXE), siendo extremadamente inseguro para transferencias corporativas.

PREGUNTA 4 - Documentación Técnica y UML

Respuesta Correcta: C (Diagrama de Secuencia)

Justificación Técnica Detallada: El Diagrama de Secuencia es la herramienta de modelado UML diseñada específicamente para visualizar interacciones dinámicas a lo largo del tiempo. En el contexto de una aplicación multiplataforma, cuando se requiere documentar "interacciones entre objetos en un escenario específico", este diagrama permite trazar el flujo de mensajes (llamadas a métodos, eventos) desde el objeto iniciador hasta los objetos receptores, respetando la secuencia temporal de ejecución. Es fundamental para entender el comportamiento del sistema antes de codificar.

Análisis de Distractores: * A) Diagrama de Clases: Muestra la estructura estática (atributos, métodos, relaciones), no las interacciones dinámicas ni temporales entre objetos en una ejecución específica. * B) Diagrama de Componentes: Se enfoca en la arquitectura de alto nivel, mostrando módulos, librerías y sus dependencias, pero no detalla el flujo de mensajes entre instancias de objetos durante un caso de uso. * D) Diagrama de Despliegue: Describe la infraestructura física (servidores, nodos, redes), no las interacciones lógicas del software a nivel de código o objeto.

PREGUNTA 5 - Metadatos y Control de Versiones Documental

Respuesta Correcta: A (Bloqueo optimista)

Justificación Técnica Detallada: El Bloqueo Optimista es el mecanismo estándar en sistemas colaborativos modernos para evitar sobrescrituras sin bloquear recursos innecesariamente. Funciona verificando si un documento ha sido modificado desde que fue leído por el usuario antes de guardar los cambios (mediante un número de versión o hash). Si la versión coincide, se guarda; si no, se alerta al usuario y se resuelve el conflicto. Esto previene las sobrescrituras mientras permite la colaboración asíncrona sin bloquear permanentemente a otros usuarios.

Análisis de Distractores: * B) Compresión ZIP: Es incorrecto porque comprimir reduce el tamaño del archivo pero no ofrece ningún control lógico sobre quién edita qué contenido ni previene conflictos de escritura simultánea. * C) Copias manuales con fecha en nombre: Es una práctica ineficiente y propensa a errores humanos ("v3_final_real.docx"). No es un mecanismo técnico de prevención, sino un parche manual que no resuelve el conflicto real durante la edición. * D) Guardado automático cada 5 minutos: Aunque evita pérdida por fallos, no controla conflictos concurrentes. Si dos usuarios guardan al mismo tiempo, uno sobrescribirá a otro sin aviso previo, perdiendo datos.

PREGUNTA 6 - Análisis de Logs y Diagnóstico

Respuesta Correcta: D (Solución propuesta para corregir el problema)

Justificación Técnica Detallada: Los sistemas de registro (logging) están diseñados para ser herramientas de observabilidad, no de resolución automática. La línea proporcionada contiene metadatos del evento (timestamp, nivel, ubicación), lo cual es vital para la auditoría forense y el diagnóstico del ingeniero. Sin embargo, el log registra el síntoma (el error ocurrido) pero nunca prescribencia la cura. La solución depende de la lógica de negocio, el contexto del estado de la aplicación y las decisiones arquitectónicas que el desarrollador debe tomar manualmente tras analizar el registro.

Análisis de Distractores: * A) Timestamp exacto: El log sí proporciona esto (2024-01-15T14:32:18.456Z), permitiendo correlacionar eventos en cascada. * B) Nivel de severidad: El log incluye [ERROR], lo cual clasifica la gravedad del incidente para priorizar la respuesta. * C) Línea específica: La notación ModuleAuth.java:127 indica con precisión dónde falló el código, facilitando la depuración (debugging).

PREGUNTA 7 - Sistemas de Archivos y Permisos ACLs

Respuesta Correcta: B (Listas de Control de Acceso - ACLs)

Justificación Técnica Detallada: Las ACLs (Access Control Lists) en sistemas ext4 (y NTFS en Windows) extienden el modelo tradicional de permisos propietario/grupo/otros. Permiten definir reglas granulares específicas para usuarios o grupos adicionales que no sean los propietarios del archivo, otorgando permisos de lectura, escritura o ejecución independientes. Esto es esencial en entornos corporativos donde la estructura organizativa no coincide exactamente con la jerarquía clásica de Unix.

Análisis de Distractores: * A) RAID: Es una tecnología de redundancia física de discos para disponibilidad y rendimiento, no tiene relación con la gestión lógica de permisos de usuarios sobre archivos. * C) Desfragmentación: Optimiza la ubicación física de los datos en el disco para mejorar la velocidad de lectura (en HDDs), pero no altera quién puede acceder a qué archivo. * D) Cifrado a nivel de disco: Protege la confidencialidad de los datos si el hardware es robado, pero una vez desbloqueado el sistema, los permisos internos siguen siendo gestionados por el usuario/grupo estándar o ACLs, no por el cifrado per se.

PREGUNTA 8 - Operadores de Búsqueda Técnica Avanzada

Respuesta Correcta: A (site:edu "Java" -forum)

Justificación Técnica Detallada: Este operador combina filtrado por dominio y exclusión de términos para refinar resultados. site:edu restringe la búsqueda exclusivamente a dominios educativos (.edu), garantizando fuentes académicas o institucionales oficiales. El operador - funciona como un "NOT", excluyendo resultados que contengan la palabra "forum". Esto elimina foros comerciales, blogs personales y sitios de soporte no verificados, dejando solo documentación técnica oficial de instituciones educativas, ideal para investigación académica.

Análisis de Distractores: * B) filetype:pdf: Filtra el formato pero no garantiza la fuente ni excluye foros. Un PDF en un foro comercial aparecería igual que uno educativo. * C) inurl:"java.com": Limita la búsqueda a un dominio específico (java.com), ignorando toda la documentación educativa alojada en universidades o repositorios académicos que no usen ese dominio. * D) "Java documentation" AND "tutorial": Es una búsqueda booleana básica que puede devolver resultados de cualquier sitio web, incluyendo foros y blogs sin verificar, ya que no filtra por tipo de dominio.

PREGUNTA 9 - Modelos de Distribución Comercial

Respuesta Correcta: C (SaaS - Software as a Service)

Justificación Técnica Detallada: El modelo SaaS describe una arquitectura donde el software se aloja en la nube y se consume como servicio vía internet, sin necesidad de instalación local. El proveedor gestiona la infraestructura, las actualizaciones, la seguridad y el mantenimiento. Esto se alinea perfectamente con el escenario descrito: "no requiere instalación local", "acceso vía navegador" y "proveedor gestiona actualizaciones". Reduce la carga operativa para la startup (DevOps externo).

Análisis de Distractores: * A) Freeware: Se refiere a software gratuito, pero generalmente implica una instalación local o ejecución en el dispositivo del usuario. No describe necesariamente el modelo de gestión en la nube ni la ausencia de instalación. * B) Shareware: Es un modelo de prueba limitada ("try before you buy") que requiere descarga e instalación. * D) Open Source: Se refiere a la disponibilidad del código fuente, no al modelo de negocio o distribución (un software puede ser Open Source y SaaS, pero el término "SaaS" describe específicamente la arquitectura de entrega descrita).

PREGUNTA 10 - Seguridad en Protocolos de Transferencia

Respuesta Correcta: B (Puerto 990 con TLS/SSL en ambos canales)

Justificación Técnica Detallada: Esta opción describe correctamente FTPS (FTP sobre SSL/TLS) en modo implícito. El puerto 990 es el estándar para la conexión de control segura de FTPS. Al usar TLS/SSL, tanto los comandos de control como los datos transferidos se cifran, protegiendo las credenciales y el contenido del tráfico de red. Es la alternativa segura al FTP tradicional cuando se necesita compatibilidad con protocolos específicos de transferencia de directorios que HTTPS no soporta nativamente de la misma manera.

Análisis de Distractores: * A) Puerto 21 + 20 sin cifrado: Describe el FTP estándar, donde las credenciales y datos viajan en texto plano, siendo vulnerables a sniffing en redes corporativas. * C) Puerto 80 (HTTP): HTTP no es seguro por defecto y no gestiona sesiones de transferencia complejas como FTP. Aunque se puede usar para subir archivos, no garantiza cifrado ni autenticación robusta sin HTTPS. * D) Puerto 443 (HTTPS): Aunque es seguro, está diseñado para web, no específicamente para transferencias FTP. Además, la opción menciona "solo para archivos de texto", lo cual es incorrecto; HTTPS puede transferir cualquier tipo de archivo binario cifrado.

PREGUNTA 11 - Gestión de Procesos y Monitorización

Respuesta Correcta: B (top / htop o Task Manager)

Justificación Técnica Detallada: Las herramientas CLI de monitorización de procesos como top, htop (Linux) o el Administrador de Tareas (Windows) están diseñadas específicamente para leer las métricas del kernel en tiempo real. Permiten visualizar qué proceso está consumiendo recursos de CPU, memoria y E/S, ordenando la lista dinámicamente por uso. Esto es esencial para identificar cuellos de botella donde una aplicación podría estar causando un bucle infinito o una fuga de memoria.

Análisis de Distractores: * A) dir / ls: Son comandos de listado de archivos en el sistema de archivos, no muestran métricas de rendimiento del procesador ni gestión de procesos activos. * C) ping / tracert: Son herramientas de diagnóstico de red (capa 3/4), útiles para verificar conectividad y latencia entre nodos, pero no miden el uso local de CPU. * D) ipconfig / ifconfig: Muestran la configuración de interfaces de red (IP, MAC, máscara), irrelevantes para el diagnóstico de consumo de recursos del sistema operativo.

PREGUNTA 12 - Estrategia de Copias de Seguridad

Respuesta Correcta: B (3 copias en 2 medios diferentes - 1 fuera del sitio)

Justificación Técnica Detallada: La regla 3-2-1 es el estándar de la industria para recuperación ante desastres (DRP). Significa: mantener 3 copias totales de los datos, almacenadas en 2 medios diferentes (ej. Disco Local + Nube o NAS), con al menos 1 copia guardada fuera del sitio físico principal (para protegerse contra incendios, robos o desastres naturales). Esta redundancia garantiza que la pérdida de un medio no resulte en pérdida total de datos.

Análisis de Distractores: * A) 2 copias en 1 medio: Incumple la regla de medios diferentes y cantidad mínima. Si el medio falla, se pierden las dos copias. * C) 5 copias en 1 servidor en la nube: Aunque hay muchas copias, todas están en "un solo medio" (la infraestructura del proveedor). Si el servicio de nube falla o es atacado, todos los datos se pierden simultáneamente (viola el principio de medios diferentes/ubicación física). * D) 1 copia local + 1 en USB portátil: Solo suma 2 copias totales y ambos medios suelen estar físicamente cerca (en la oficina), fallando el criterio de "fuera del sitio".

PREGUNTA 13 - Formatos de Archivo y Preservación Digital

Respuesta Correcta: D (.pdf/a)

Justificación Técnica Detallada: El estándar PDF/A (ISO 19005) está diseñado específicamente para la preservación digital a largo plazo. A diferencia del PDF normal, PDF/A incrusta todos los recursos necesarios (fuentes, metadatos, perfiles de color) dentro del archivo, eliminando dependencias externas que podrían dejar el documento ilegible en el futuro si se pierden las fuentes o software específico. Es el formato preferido para archivos legales y técnicos institucionales.

Análisis de Distractores: * A) .docx: Aunque es un estándar abierto (Office Open XML), sigue dependiendo de la interpretación del renderizador de Microsoft u otros, y puede tener problemas de compatibilidad visual a décadas vista si las especificaciones cambian. Además, el formato binario original de Office (.doc) es propietario. * B) .odt: Es un formato abierto excelente para edición, pero menos universal que PDF para la visualización definitiva. Para preservar "cómo se ve" el documento exactamente, PDF/A es superior. * C) .rtf: Es un formato antiguo con limitaciones en soporte moderno de fuentes y diseño complejo. No tiene estándares de preservación robustos como ISO 19005.

PREGUNTA 14 - Protocolos de Mensajería y Encriptación E2EE

Respuesta Correcta: C (Encriptación de Extremo a Extremo - E2EE)

Justificación Técnica Detallada: La E2EE garantiza que los datos se cifran en el dispositivo del remitente y solo se descifran en el dispositivo del receptor. Las claves privadas nunca salen de los dispositivos finales, lo que implica técnicamente que ni siquiera el servidor intermediario (el proveedor del servicio) tiene la capacidad criptográfica para leer el contenido de los mensajes. Esto cumple estrictamente con el requisito de "ni siquiera el proveedor pueda leer".

Análisis de Distractores: * A) TLS/SSL en transporte: Cifra la conexión entre el cliente y el servidor, pero una vez que el mensaje llega al servidor para ser reenviado, se descifra aquí. El proveedor del servicio podría acceder a los datos en este punto (ej. Google puede leer correos SMTP no E2EE). * B) Autenticación OAuth2: Es un protocolo de autorización para permitir acceso seguro a APIs, no un método de cifrado de contenido de mensajes. * D) Firewall perimetral: Protege la red contra accesos externos, pero no encripta el contenido interno que viaja entre usuarios legítimos del servicio.

PREGUNTA 15 - Mantenimiento Preventivo y SSDs

Respuesta Correcta: B (Desfragmentación periódica)

Justificación Técnica Detallada: La desfragmentación es una técnica diseñada para discos duros mecánicos (HDD) donde los datos se almacenan en pistas magnéticas. Al mover fragmentos de archivos contiguamente, se reduce el movimiento del cabezal de lectura y mejora el rendimiento. En SSDs, no hay partes móviles; el acceso a cualquier celda es instantáneo. Además, la desfragmentación genera escrituras innecesarias que consumen los ciclos de escritura limitados del SSD, reduciendo su vida útil. Los sistemas modernos detectan esto automáticamente y ejecutan TRIM en lugar de defrag.

Análisis de Distractores: * A) Actualización de firmas antivirus: Siempre necesaria para proteger contra malware, independientemente del tipo de disco. * C) Verificación de espacio en disco: Útil para evitar fallos por falta de espacio en el sistema operativo, aplicable a cualquier almacenamiento. * D) Limpieza de archivos temporales: Mejora el rendimiento general y libera espacio, recomendado tanto para HDD como SSD.

import ftplib
import os
import sys
from dotenv import load_dotenv  # Requiere instalar python-dotenv

# Cargar variables de entorno desde .env (seguridad mejorada)
load_dotenv()

def transferir_archivo(archivo_local, ruta_servidor):
    """Transfiere un archivo a servidor FTP con manejo seguro de errores"""

    HOST = os.getenv('FTP_HOST', 'ftp.empresa.com')
    # Obtener credenciales desde entorno en lugar de hardcodear
    USUARIO = os.getenv('FTP_USER')
    CONTRASEÑA = os.getenv('FTP_PASSWORD')

    if not USUARIO or not CONTRASEÑA:
        print("Error: Credenciales no encontradas en variables de entorno.")
        return False

    try:
        ftp = ftplib.FTP(HOST, timeout=10)

        # Login seguro (las credenciales vienen de memoria, no del texto plano)
        ftp.login(user=USUARIO, passwd=CONTRASEÑA)

        # Intentar cambiar directorio, crear si no existe
        try:
            ftp.cwd(ruta_servidor)
        except ftplib.error_perm:
            print(f"Directorio {ruta_servidor} no existe. Creando...")
            ftp.mkd(ruta_servidor)
            ftp.cwd(ruta_servidor)

        # Extraer solo el nombre del archivo para subirlo al servidor
        nombre_archivo = os.path.basename(archivo_local)

        with open(archivo_local, 'rb') as archivo:
            # Corregido: Pasar string con nombre remoto y objeto de archivo
            ftp.storbinary(f'STOR {nombre_archivo}', archivo)

        print("Transferencia exitosa.")
        return True

    except ftplib.error_perm as e:
        print(f"Error de permisos FTP: {e}")
        return False
    except ConnectionRefusedError:
        print("No se pudo conectar al servidor FTP. Verifique HOST y red.")
        return False
    except FileNotFoundError:
        print(f"El archivo local '{archivo_local}' no existe.")
        return False
    except Exception as e:
        # Logging de error genérico (en producción usar logging module)
        print(f"Error inesperado: {e}")
        return False

if __name__ == "__main__":  # Corregido: doble guion bajo al final
    # Validación básica de archivos antes de ejecutar
    if os.path.exists("documentacion.pdf"):
        transferir_archivo("documentacion.pdf", "/public/docs")

El código original presenta un diseño frágil basado en atributos públicos directos y setters básicos. Para una aplicación DAM profesional, debemos aplicar los principios de encapsulamiento, inmutabilidad donde sea posible y tipado estricto.

from datetime import datetime
from enum import Enum, auto

# Patrón: Enum para estados definidos y seguros
class EstadoDocumento(Enum):
    BORRADOR = "Borrador"
    REVISION = "En Revisión"
    APROBADO = "Aprobado"
    PUBLICADO = "Publicado"

# Patrón: Clase Separada para Versiones (Single Responsibility)
class Version:
    def __init__(self, major=1, minor=0):
        self.major = major
        self.minor = minor

    def increment(self, minor_only=True):
        """Incrementa la versión según estrategia SemVer"""
        if minor_only:
            self.minor += 1
        else:
            self.major += 1
            self.minor = 0
        return f"{self.major}.{self.minor}"

class DocumentacionTecnica:
    def __init__(self, titulo, autor="Desconocido"):
        # Validación en constructor (Fail Fast)
        if not isinstance(titulo, str) or len(titulo.strip()) == 0:
            raise ValueError("El título no puede estar vacío")

        self._titulo = titulo.strip()
        self._autor = autor.strip() if isinstance(autor, str) else "Desconocido"

        # Metadatos de trazabilidad (RA7 - Documentación Técnica)
        self.version_actual = Version(1, 0)
        self.fecha_creacion = datetime.now()
        self.fecha_ultima_modificacion = None
        self.estado = EstadoDocumento.BORRADOR

    @property
    def titulo(self):
        return self._titulo

    @titulo.setter
    def titulo(self, valor):
        """Setter con validación de integridad"""
        if not isinstance(valor, str) or len(valor.strip()) == 0:
            raise ValueError("El título debe ser un string no vacío")

        # Registrar hora de modificación para auditoría
        self.fecha_ultima_modificacion = datetime.now()
        self._titulo = valor.strip()

    @property
    def autor(self):
        return self._autor

    @autor.setter
    def autor(self, valor):
        if not isinstance(valor, str) or len(valor.strip()) == 0:
            raise ValueError("El autor debe ser un string no vacío")
        self._autor = valor.strip()

    def actualizar_version(self, minor=True):
        """Gestiona la evolución del documento"""
        nueva_versao = self.version_actual.increment(minor)
        self.estado = EstadoDocumento.REVISION # Cambia estado automáticamente
        return f"Versión actualizada a {nueva_versao}"

    def aprobar_documento(self):
        """Valida y cambia el estado del documento"""
        if self.fecha_ultima_modificacion is None:
            raise ValueError("Debe haber modificaciones antes de aprobar")
        self.estado = EstadoDocumento.APROBADO

    # Patrón: Representación legible para logs/diagnóstico
    def __str__(self):
        return (f"=== Documento Técnico ===\n"
                f"Título: {self._titulo}\n"
                f"Autor:  {self._autor}\n"
                f"Versión: {self.version_actual.major}.{self.version_actual.minor}\n"
                f"Estado:  {self.estado.value}\n"
                f"Creado:  {self.fecha_creacion.strftime('%Y-%m-%d %H:%M')}\n"
                f"Modif.:  {self.fecha_ultima_modificacion.strftime('%Y-%m-%d %H:%M') if self.fecha_ultima_modificacion else 'Nunca'}")
